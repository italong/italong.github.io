2.面向对象的特点

答：首先面向对象是指，将一个个事务抽象成为一个个行为，这样做的目的不是为了去关注它一个个的行为，而是为了避免不必要的公共部分，提高代码的复用性；面向对象有三大特性：封装、继承、多态；

封装是指将一个对象的属性私有化，同时提供给能给外界访问的get/set方法；继承则是说子类继承父类，它拥有父类的所有属性和方法，但是无法访问父类的私有属性和方法，它可以用自己的方式实现父类的方法，同时它也拥有自己的属性和方法；多态是指引用变量在编程时，无法知道它引用的数据类型，和它引用的到底是哪个方法；只有在运行时才能知道，也就是说该引用变量到底指向哪个类的实例对象，该引用变量发出的方法到底调用哪个类实现的方法；多态的实现主要分为两种，重载和重写，重载呢是说编译时多态，重写呢则是运行时多态；重载是说和（）拥有相同的返回类型，但是它可能参数类型不同，参数个数不同或者两者皆不同；重写则是说，和其方法返回类型不同，同时不能抛出更多异常，要比原方法更高效；

3.抽象与接口的区别

答：第一，接口的方法默认是public，同时方法没有具体实现，但是从1.8开始，方法有默认实现；而抽象类可以有非抽象方法；第二，接口中变量除了static （final）之外，不能有其他变量，而抽象类中则不一定；第三，接口的默认修饰符是public，而抽象类的则是dafult public protect，而它不能使用private是因为它本身就是为了被重写才定义的；（第四，一个类可以实现多个接口，但只能实现一个抽象类，接口本身可以通过extends关键字扩展多个多个接口）；第五，抽象是对类的抽象，是一种模板设计；接口是对行为的抽象，是一种行为规范；

4.==和equals的区别

5.hashmap的put方法的具体流程

3.判断当前table[i]==null，为空则直接插入；

4.不为空则需要先判断该位置是否为红黑树,如果是则走红黑树的put流程；若不是，则遍历链表，在链表中若存在key相同的值，则直接覆盖，若不存在，则需要在链尾判断next是否为空，为空则添加，会判断当前链表长度是否大于8，若大于则需要转化为红黑树，走红黑树的put流程；

2.初始化后扩容一般是扩容2倍；

3.扩容之后先新建数组，遍历老数组，先判断hash值模与newMap-1是否为0，若是则直接添加，不是则判断是否为红黑树，若是则走红黑树添加；若不是则遍历链表，可能需要拆分链表，判断hash值&oldMap是否为0，若是则newMap[j] = oldMap[j]，若不成立，则赋给newMap[j+oldMap] ；

1.线程创建的方式？

2.线程包括哪些状态？

答：首先创建线程 NEW状态，之后变为可执行状态RUNNABLE,最后是终止线程状态TEMINATED；而在可执行状态后存在又可分三种状态，第一是阻塞BOLOCK，只有获取锁才能进入RUNNABLE;第二是WAITING等待，只有调用notify方法才能进入RUNNABLE；第三是TIMED_WAITING时间到就会获取；

答：通过调用join方法，比如三个线程，线程二中通过trycatch调用t1的join方法只有t1运行完t2才能运行，t3同理；

4.notify()和 notifyAll()有什么区别？

答：notify只会随机唤醒某一个wait线程；而notifyall则会唤醒所有wait线程；

5.在 java 中 wait 和 sleep 方法的不同？

答：首先他们都是让线程暂时放弃CPU的执行权，进入阻塞状态；而他们的不同大致在三个方面：方法归属，醒来时机和锁特性不同；第一方法归属上，sleep（long）是Thread类的静态方法，而wait则是object类的方法，每个对象都有；第二醒来时机上，执行sleep带参和wait带参都会在等待数秒后醒来；wait带参和wait可以被notify方法唤醒；wait方法如果不被唤醒就会一直执行下去；它们都可以被打断执行；第三锁特性上，wait方法执行一定要带锁也就是说一定要和syronized或lock一起用；wait执行时，允许其他线程获取该对象锁；sleep方法执行时，不允许其他线程获取对象锁；

6.如何停止一个正在运行的线程？

答：有三种方法：第一通过退出标志，也就是说定义一个boolean的值，通过改变值，来控制线程的运行；第二种是已经作废的方法stop；第三种则是interupt方法，本质上也是退出标志；

4.缓存高并发常见场景

4.1 缓存穿透

概念：某些高级攻击，通过频繁发送不存在的数据请求，当服务从Redis中无法查询到时，就需要去查询数据库，之后返回结果，如果这时候有大量请求打到数据库，这时候就可能造成数据库宕机；

解决方案：

1.在缓存中存放大量的空值key，但是一般不推荐这样使用

2.通过布隆过滤器，在数据查询缓存前，我们先经过布隆过滤器过滤，如果存在再走缓存，如果不存在再去查询数据库，将数据缓存，而数据则一直在布隆过滤器重试；从而达到过滤的作用，而布隆过滤器的底层是散列函数通过三次hash来判断是否存在这个值，不过它存在一定的误判率，一般我们通过控制数组长度来将误判率控制在5%以下；

4.2 缓存击穿

概念： 某个key到了过期时间后，大量当前key的请求发送过来，查询缓存发现已经过期后，直接查询数据库，会给数据库造成很大的压力；

解决方案：

  1. 互斥锁：我们通过增加互斥锁，假如说一个线程1请求当前key，发现过期，这时候它会加互斥锁，再查询数据库去重建缓存数据，这时候如果线程2进来，他也去缓存中查询这个key，它是无法操作的，只能不断休眠重试获取互斥锁；当线程1操作得到缓存后，释放了互斥锁，线程2就查询缓存，从而缓存命中；一般来说和钱有关的我们就使用这种方案，不过它的性能可能比较差。
  2. 逻辑过期：我们通过设置逻辑过期，假如说当前线程1来获取缓存，发现过期了，我们这时候也获取互斥锁，成功后，再开启一个线程2，通过线程2来查询数据库重建缓存，在这个过程中，线程1不会等它完成，而是直接返回过期了的缓存数据；而如果再线程2重建缓存的这个过程，线程3来了，它查询也发现key过期了，同时获取互斥锁失败，它就会直接返回过期数据；这种方案我们我们一般就使用在对性能要求较高的场景；它保证了高可用性；

4.3 缓存雪崩

概念：和缓存击穿不同，缓存雪崩就是说大量key值过期或者Redis宕机，这时候大量不同数据请求发送过来，同时查询缓存，发现过期，需要查询数据库重建缓存，这时候数据库就有可能压力过重而雪崩；

解决方案：

  1. 通过给不同的Key的TTL设置随机值；
  2. 通过Redis集群来提高服务的可用性；哨兵模式、集群模式等（宕机的话）
  3. 通过给缓存业务添加降级限流策略；Nginx/Gateway（降级可作为系统的保底策略 如穿透 击穿 雪崩）
  4. 给业务添加多级缓存； 通过Guava（gerufa） 或者 Caffeine(comfiin)

5.分布式锁场景

应用场景：抢券 定时任务 幂等性场景

定时任务：文章延时任务准时发布

答：首先在我们选择延迟准时发布后，会将定时任务，发送到redis中，同时在MySQL中生成任务数据，我们先判断当前时间是否大于执行时间，如果成立则发送给Redis中的list中异步进行任务队列发送，如果不成立则判断执行时间是否小于预设时间，成立则到zset中生成未来任务，然后定时每一分钟刷新任务到list中。

主要是通过Redis的setnx特性完成分布式锁的功能，A锁获取到锁以后其他客户端不能进行操作，只能等待A锁释放；

抢券：分布式锁

答：一般来说，单体情况下抢卷流程一般是这样的，首先线程1获取互斥锁，获取成功，查询库存，有就扣减库存，没有就抛异常；当线程2来时，它也获取锁，线程1还没有完成，只能一直循环获取锁，直到线程1释放他才能获取成功；这是一般单体流程；而分布式的区别在于经过Nginx的反向代理和负载均衡后，它会有多个端口同时去获取，这时候我们就要保证怎么使他们获取到的是同一把锁；所以我们需要分布式锁，大致流程也是一样：先需要获取锁，一般我们是通过Redssion的setnx命令来进行操作，获取锁成功后，执行相关业务，这里会有一个问题，第一个就是我们如果不设置超时时间，那么如果Redis在获取锁成功后Redis宕机了或者业务超时了，那么这样就有可能造成死锁问题，这里主要的机制有两种，一种呢是我们自己评估，设定合理的过期时间；另一种呢是我们是采取一个看门狗机制，每次一定时间后刷新它的过期时间，公式是relasetime/3的间隔时间，同时加锁和看门狗机制一般都是配合lua脚本使用可以有效保证它的原子性，也就是说要么同时成功，要么同时失败；不过一般业务执行都是很快的都是ms级的；

6.Redis集群

6.1 主从复制（保证高并发)

答：是这样的，Redis单节点并发能力有限，于是我们采取了主从复制，一般来说就是一主多从，一方面提高了并发量，另一方面也实现了读写分离，一般主节点负责写，从节点负责读；而主从复制又分为全量同步和增量同步，我们先说一下全量同步，第一，我们从节点发送复制请求并携带offset和replic id给主节点，建立连接；第二，会比对是否是第一次请求，也就是比较replic id是否一致，不一致，则需要同步，主节点发送自己的offset和replic id给从节点；第三，同时主节点执行bgsave命令，生成RDB文件后，交给从节点去执行，从节点清空自己的数据，然后执行主节点发送的RDB文件，这样就保持了一致了；当然如果RDB文件生成执行的时候，主节点又收到请求，主节点就会以命令的方式记录到缓冲区，缓冲区本质是一个日志文件，最后把这个日志文件发送给从节点，这样也就保证了最终的数据一致了；后续再同步就需要依赖这个日志文件了；

增量同步是指当从节点服务重启之后，数据就不一致了，所以这个时候，从节点就会请求节点同步数据，这时候主节点还是先判断是不是第一次同步，不是就获取从节点offset偏移量，和主节点进行比对，主节点比从结点多了多少偏移量，我们就以命令的形式同步这部分的数据；

6.2 哨兵模式（保证高可用）

答：是这样的，哨兵模式主要是实现主从集群的自动故障修复（监控 数据恢复 通知），我们当时设置的redis参数一般是1主（8万）1从（10万）+哨兵，已经基本保证了我们项目的基本需求；而集群脑裂，主要指的是当前集群中，由于主节点和从节点位于不同的网络分区，使得sentinel没有能够心跳感知到主节点，所以通过选举的方式提升一个从节点为主节点，这样就存在两个master就像大脑分裂了一样，这样会导致客户端还在老的主节点传输数据，等网络恢复后，它就会降为从节点，而它接收到的数据呢也会丢失，我们一般采取修改Redis的两个配置：1.通过设置最少从节点数量；2.通过缩短主从数据的延迟时间；只要不满足要求，我们就会拒绝请求，从而让这部分数据没有丢失；

6.3 分片集群（海量数据存储）

答：分片集群就是说将数据分段进行存储，每个master之间存储不同的数据；

第二就是说每个master下有多个slave节点；

1.索引

1.什么是索引？

答：知道一些的，索引是帮助mysql提高查询数据效率的数据结构，它可以提高数据查询效率，降低数据库IO成本；同时，可以提高数据排序速度，帮助数据库排序，从而减少cpu的压力；

2.索引的数据结构？

答：索引底层采用的B+树，它相对于B树来说，B树的非叶子节点和子节点都会存储数据，而B+树则只有叶子节点存储数据；同时，它阶数更短，路径更低；在叶子节点上采取双向链表的方式，可以快速扫库和区间查询；

2.存储引擎

MYSQL支持的存储引擎？

答：InnoDB存储引擎是mysql5.5以后默认的存储引擎，它支持事务、外键、行级锁和表级锁；

MyISAM是早期的存储引擎，不支持事务，只有表级锁，也没有外键，用的不多；

Memory主要把数据存储在内存，支持表级锁，没有外键和事务，应用也不多；

存储引擎在mysql体系结构哪一层，主要特点？

答：

答：MYSQL主要分为连接层服务层引擎层存储层；连接层主要存放JDBC ODBC连接工具；服务层则是一些缓存SQL解析器、连接池等；引擎层存放存储引擎；存储层则是一些系统文件和其他文件、日志；

InnoDB是一种兼具高可靠性和高性能的存储引擎；从5.5开始mysql默认的存储引擎，它支持事务（ACID）外键（保证数据的完整性和可靠性）行级锁（支持并发访问性能）表级锁；ibd后缀文件(表空间文件) frm文件后缀（表结构文件）

什么是聚簇索引和非聚簇索引？回表？

答：聚簇索引是指将主键和数据都存放在叶子节点，其存储的是整行数据；而非聚簇索引是指将主键和数据分开存储，一般我们自己定义的索引就是非聚簇索引；回表是指通过二级索引得到对应的主键值，再到聚集索引中查询得到整行数据，这个过程就是回表查询；

什么是覆盖查询？mysql超大分页怎么处理？

答：覆盖查询是指使用了索引在返回的列中全部都能查询得到，比如说我们查询条件人名，我们以人名创建了索引，那么我们返回的列中必须只有人名和id；mysql超大分页主要指的是通过limit进行分页处理时，最开始数据量可能不大，当达到百万级时，通过limit查询会减少它的查询效率，因为它每次查询得到的结果都需要将前面的都查询出来，然后丢弃，一般我们采取覆盖查询+子查询的方式优化，通过覆盖查询，先查询出id，再通过id来获取得到该字段的数据

索引创建原则？

答：一般创建索引式需要在数据量很大的情况下去创建的，像是数据量达到了10w条以上，或者说该字段查询很频繁的情况我们就需要创建索引；尽量选择区分度高的创建唯一索引，这样使用索引的效率就越高，如果式字符串类型，字段的长度较长，我们就可以创建前缀索引；当然并不是所有都应该创建索引，我们也要控制索引的数量，因为增加索引也会导致新增加的速度变慢；

什么时候索引会失效？

答：失效的场景有很多，我说一些我具体遇到的失效场景，像是返回的列没有能在覆盖索引中全部查出，也就是跳过了索引的列没有满足最左匹配原则，又或者范围查询右边的列，在索引列上进行一系列运算操作；字符串不加单引号（查询优化器会优化将其类型转化） ，模糊查询以%开投也会造成索引失效；一般我们通过explain来查看具体的效果和优化方案；

2.事务

事务是什么？

答：事务时一组操作的集合，它会将所有的操作一起向系统发送提交或撤销的请求，也就是说要么都成功要么都失败；

事务的特性详细说一下？

答：事务具有四大特性：原子 一致 隔离 持久 ACID：我举一个例子具体说明一下：A向B转账500，原子性体现在转账提交成功均一起提交；一致性体现在A转500 B收500，总额一致；隔离性则是转账过程中不受其他事务影响；持久性则是说最后完成后数据持久化也就是落盘；

并发事务带来哪些问题？怎么解决 ？事务的隔离级别？

答：

undo log 和redo log 的区别？

答：redo log 记录的是数据页的物理变化，当数据库宕机后可以同步数据一般是通过redo log buffer 和 redolog file来操作，当它大量的更改删除操作时，我们先在缓冲池中进行操作，存到redo log buffer中再同步到磁盘的redo log file文件中，这时候如果宕机了，我们之后只需要通过redo log file 来恢复数据就可以了，这里并不是进行一个多余的操作，是为了将之前杂乱的IO能够有序化处理，保证了事务的原子性；而undo log则是逻辑日志，用于记录数据被修改前的数据，也就是说如果有一个delete就在undo log中有一个insert语句，用来进行回滚操作，一般用于MVCC中，保证了事务的一致性和持久性；

事务中的隔离性如何保证的呢？

答：一般通过加锁（排他锁）和MVCC多版本并发控制来进行具体操作，排他锁指的是一个事务对该行加排他锁之后，其他事务不能获取该行的其他锁；

3.多版本控制

解释一下MVCC？

答：事务的隔离性主要由锁和MVCC保证：

MVCC主要由三个部分组成，隐藏字段，undo log日志，readview视图；隐藏字段主要分为两个，事务id，回滚指针；事务id主要是记录每一次操作的事务id，它是自增的；回滚指针则是指向上一个版本的事务版本记录地址；undo log日志主要保存的是回滚日志和版本链；回滚日志主要存储老版本数据，而版本链则是在多个事务进行并行操作某一行数据的时候，记录不同事务修改数据的版本，通过回滚指针形成一个链表；readview读视图主要是解决一个事务查询选择版本的问题，它由自己的匹配规则同时根据当前的一些事务id判断该访问哪个版本的数据；如果是RC，则是每一次快照读时都要生成readview，而RR则是第一次快照读时生成后续复用就可以了；

4.乐观锁和悲观锁

答：悲观锁：

悲观锁认为被它保护的数据是极其不安全的，每时每刻都有可能被改动，⼀个事务拿到悲观锁后，其他任何事务都不能对该数据进⾏修改，只能等待锁被释放才可以执⾏。数据库中的⾏锁，表锁，读锁，写锁均为悲观锁。

乐观锁：乐观锁认为数据的变动不会太频繁。乐观锁通常是通过在表中增加⼀个版本(version)或时间戳(timestamp)来实现，其中，版本最为常⽤。事务在从数据库中取数据时，会将该数据的版本也取出来(v1)，当事务对数据变动完毕想要将其更新到表中时，会将之前取出的版本 v1 与数据中最新的版本 v2 相对⽐，如果 v1=v2，那么说明在数据变动

期间，没有其他事务对数据进⾏修改，此时，就允许事务对表中的数据进⾏修改，并且修改时 version会加 1，以此来表明数据已被变动。如果，v1 不等于 v2，那么说明数据变动期间，数据被其他事务改动了，此时不允许数据更新到表中，⼀般的处理办法是通知⽤户让其重新操作。不同于悲观锁，乐观锁通常是由开发者实现的。
